# Plan 003 — the MVP (npm-first) + architecture decisions

> Tech-lead pass over the confirmed specs, market research, and the Plan 002 dry-run. **Architecture answers are recommendations awaiting Victor's sign-off; the MVP plan is built on them.**

---

## TL;DR

- **You need a service — but not a database.** The rejected thing (a precious stateful graph DB someone must nurse) and the needed thing (a stateless, rebuildable query/aggregation surface) are different objects. Build the second, never the first.
- **Truth lives in repos as committed CycloneDX SBOMs. The index is derived, disposable, rebuildable.** MVP index = an in-memory graph snapshotted to a single **SQLite file**. Delete it, rebuild from source, lose nothing.
- **The core is a mode-agnostic library.** Wrap it as a CLI (serverless / air-gapped) and as a thin service (org-scale). Same artifacts, same index format; the only variable is who builds the index and when.
- **Consume OSV. Produce/consume CycloneDX. npm only. Sell "exposed," never "breaks."**

# Part 1 — Architecture decisions

## a. Do we need a central backend/service? — YES, with a hard distinction

The owner rejects a **precious stateful database** (Neo4j/Postgres that is the source of truth and must be provisioned, backed up, kept alive). Correct — drop the v0.1 stack for the MVP.

But the compliance question is inherently cross-org ("every product exposed to CVE-X across the org, with owners"), and you can't answer that without collecting the union of every repo's dependencies somewhere. So an **aggregation/query surface is unavoidable** — but it's a **derived index, not a source of truth**: wipe it and rebuild from source in seconds. The dry-run proves the scale is trivial (18 KB / 89 packages, ~1 ms; low-MB and sub-second even at Siemens scale). **Build the service as a rebuildable index, not a database** — fully consistent with the owner's constraint.

## b. What storage looks like — committed CycloneDX SBOMs (truth) + one SQLite file (index)

- **Source of truth:** per-repo committed **CycloneDX SBOM** (`.bojji/sbom.cdx.json`) — git-versioned, keyless, in-perimeter, rebuildable by definition.
- **Index/query surface:** a **single SQLite file** — a *file*, not a server; no provisioning, trivial backup, embeddable in both CLI and service. The graph lives in RAM at runtime; SQLite is just the snapshot.
- **Rejected for MVP:** Neo4j/Postgres/Redis (the "keep-alive DB"). **In pocket for later:** DuckDB (analytical scans), an "index repo" collection pattern (v1.1), object storage for the push path.
- **File-per-repo vs one index repo:** source artifact = file committed in each repo (self-describing, keyless, matches C3/C4); collection for MVP = read those files; the "one index repo" is a clean v1.1 upgrade.

## c. Can Bojji work WITHOUT a central backend? — Yes, three modes off one core

Build the index-builder as a **library**, then run it three ways:

| Mode | Cross-org query | Freshness | Who runs it | Fits |
|---|---|---|---|---|
| **On-demand CLI / GitHub Action** | Slow (reads every repo per query) | Live | Each engineer | Small team (<~50 repos); the free, viral entry |
| **Pre-built static index** (scheduled) | Fast | Stale to rebuild cadence | A scheduled job | Air-gapped / high-security (+ offline OSV mirror) |
| **Thin always-on service** | Fast | Fresh (webhook) | An ops-owned process | Siemens-scale (thousands of repos) |

**The unifying insight:** all three share the same SBOM artifacts and the same index format — the service is just one deployment of a library that also runs as a CLI. Build the library first.

## Other open decisions (decided by the tech lead — proposed)

- **"Product" = a releasable/deployable unit that ships with a version** (the CRA's "product placed on the market") — not a repo, not an npm package. Declared in `product.yaml`, defaulting to the repo/publishable package. In the ontology it's the `Product` node that exposure and ownership attach to.
- **SBOM format = CycloneDX only** for MVP (mature buildless npm tooling; `purl` = the canonical identity key; VEX is first-class → the standard way to say "exposed but not affected"). Keep the model format-neutral so SPDX import is a later adapter, not a rewrite.
- **Vulnerability feed = OSV (osv.dev), consumed not built.** Package-version-range native, aggregates GitHub Advisory, downloadable dumps you can mirror for air-gapped use. (NVD's CPE matching is painful; GHSA already inside OSV.)
- **Minimal v1 ontology:**
  - **Entities:** `Product`, `Package` (external dep, keyed by `name@version` / purl), `Repository`, `Team`, `Person`.
  - **Relationships:** `Product —produced_from→ Repository`; `Product —depends_on→ Package` (`direct|transitive`); `Package —depends_on→ Package`; `Product —owned_by→ Team`; `Team —contact→ Person`.
  - **Defer:** Service, Artifact, Documentation, Release history, incidents, deployments — add only when a real query pulls for them.

# Part 2 — the npm-first MVP

## Goal & the one demo

Point Bojji at an npm org (a few repos incl. a SWWC v2 **clone**) and answer owned, org-wide exposure in one call:

```
bojji expose GHSA-wf5p-g6vw-rhxx        # an axios advisory
→ 3 products across 2 teams exposed via axios@1.5.0
  • swwc-v2   team: Design Systems  contact: …  path: swwc-v2 → axios            [high]
  • app-web   team: Web             contact: …  path: app-web → @acme/ui → axios  [high]
  • app-admin team: Web             contact: …  path: app-admin → axios           [medium — owner guessed]
```

The value moment: the answer is **owned and routed**, not "a list of repos." That's what the ontology buys and what compliance needs.

## The per-project files (keyless, local-first)

- `.bojji/sbom.cdx.json` — CycloneDX SBOM, generated buildless from the lockfile (machine truth).
- `.bojji/product.yaml` — the descriptor (human part, C3): product name, owning team, contacts; auto-filled from CODEOWNERS / `package.json` author, gaps recorded, never blocking.

Every node/edge carries provenance + confidence + last_seen; confidence stays 3-level `high|medium|low` for MVP.

## Ingestion

- **On adopt (C3):** `npx bojji init` → writes the SBOM, scaffolds `product.yaml` (auto-fills owner from CODEOWNERS), records gaps, never blocks.
- **At release (C4):** CI regenerates + commits the SBOM. **MVP collection = the service reads repos with its own read-only access** (avoids building OIDC in week one). **Keyless OIDC push = v1.1.**

## Query surface

| Surface | In MVP | Deferred |
|---|---|---|
| REST | `/exposure?advisory=`/`?package=`, `/products?exposed_to=`, `/products/{id}` | full `/graph`, health scores |
| MCP | `find_exposure`, `find_owner`, `find_dependents` | `impact_analysis`, `search_documentation`, `project_health` |
| UI | one page: search + exposure table (product, team, contact, path, confidence, last_seen) | graph view, AI chat, dashboards |

## Exposure feature, end to end

1. **Collect** every `.bojji/sbom.cdx.json` → in-memory graph (nodes keyed by purl) → snapshot to SQLite.
2. **Overlay** OSV npm advisories (API online; downloaded dump when air-gapped) using a **vetted semver library** (the prototype's toy comparator must be replaced).
3. **Query:** advisory/package → affected `Package` nodes → reverse-traverse to `Product`s → join owner/contact → return with **path + confidence + provenance**.
4. **Honest verb:** label "exposed", never "breaks." Read a committed **VEX** file to mark `not_affected` (VEX authoring UI deferred).

Fix the two dry-run caveats here: separate `publishes` from `depends_on` in reverse traversal; key `Package` by `name@version` (multiple concurrent versions are real).

## Architecture topology (MVP)

- Source of truth: per-repo committed CycloneDX SBOM + `product.yaml`.
- Core: a **mode-agnostic index-builder library** in TypeScript/Node.
- MVP runtime: **one small stateless Node service** — reads repos read-only, builds the graph in memory, snapshots to a single SQLite file, serves REST + MCP + the one UI page; re-index on webhook/interval. Same core ships as a `bojji` CLI.
- No Neo4j / Postgres / Redis / BullMQ.

## Milestones (solo dev)

- **M0 (wk 1) — thinnest slice:** harden the prototype to emit/read CycloneDX + real semver, run over the SWWC v2 clone + 2–3 repos locally, one live OSV lookup, print exposed products from the CLI. Proves the two riskiest unknowns (real lockfile parses; OSV range-matching works).
- **M1 (wk 2–3):** `product.yaml` + `bojji init` (owner from CODEOWNERS); ownership in output; SQLite snapshot + rebuild.
- **M2 (wk 4–5):** the stateless service — REST + read-only collector across N repos.
- **M3 (wk 5–6):** MCP server (3 tools) + the one UI page.
- **M4 (wk 7):** air-gapped OSV mirror + CI SBOM-refresh Action; polish the demo (keyless OIDC push begins as v1.1).

## Out of scope for the MVP

Multi-ecosystem (npm only); graph DB; AI chat; interactive graph viz; Jira/Confluence/GitLab connectors; multi-tenant SaaS; keyless OIDC push (read-only collector first); VEX authoring UI; "what breaks"/code-usage analysis; SPDX; license compliance; Service/Documentation/Artifact entities; numeric confidence; real-time freshness guarantees.

## Top risks

1. **GitHub already gives orgs a dependency graph + Dependabot + SBOM export for free** — the sharpest honest risk, bigger than Moderne for the small-team entry. **De-risk:** don't compete on GitHub's turf; lean hard into on-prem / air-gapped / non-GitHub (self-hosted GitLab), product-level ownership routing, and the no-DB cost story. Pitch the demo in that frame, not as "a nicer dependency graph."
2. **"Exposure" credibility trap** — strict verb discipline, always show the path, read committed VEX.
3. **Transitive/nested versions & monorepos** — key by `name@version`/purl; fix reverse-traversal.
4. **OSV semver-range accuracy** — use a vetted library + OSV's own ranges; never hand-roll.
5. **Freshness at scale** — SBOM committed → webhook re-index; show `last_seen` on every answer.
6. **Cross-source identity/merge** — npm-first collapses this (package identity from lockfile, product identity from `product.yaml`); defer general entity resolution.
7. **Solo-dev scope creep** — the mode-agnostic core + the ruthless out-of-scope list; every deferred item has a named later horizon.
